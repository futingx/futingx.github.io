# 项目初始化和执行流程指南

本文档整合了 **Claude Code** 推荐的项目启动与开发流程，涵盖从初始化到提交的全生命周期管理。遵循这套流程，可显著减少返工，提升代码质量与团队协作效率。

---

## 一、项目初始化（6 个习惯性动作）

在开始任何新项目之前，请按顺序完成以下 6 项设置，为后续开发打下稳固基础：

| 步骤 | 动作 | 说明 |
|------|------|------|
| **Step 1** | **项目初始化** | 向 AI 清晰描述项目目标（用途、设备、功能、页面、设计风格），由 AI 生成项目骨架（目录结构、基础配置文件）。 |
| **Step 2** | **建立项目上下文** | 运行 `./init` 命令（或手动创建）让 AI 自动生成 `CLAUDE.md` 文件，该文件记录项目背景、技术栈、编码规范等核心信息。 |
| **Step 3** | **配置权限与默认模式** | 编辑 `.claude/settings.json`，设置权限规则；复杂项目建议将默认模式设为 `plan`，强制先规划后执行。 |
| **Step 4** | **功能开发** | 一次只开发一个功能，严格遵循 **Explore → Plan → Implement** 的节奏（详见第二节）。 |
| **Step 5** | **代码审查与测试** | 使用 `/review` 命令让 AI 自动生成测试用例并执行，确保功能正确性。 |
| **Step 6** | **提交代码** | 使用 `git commit` 保存进度，AI 可协助生成规范的 commit message。 |

---

## 二、四阶段工作流（核心执行循环）

每个功能或任务都应按以下四个阶段循环推进，一轮结束后回到 Explore 开始下一个任务。

```
Explore（探索） → Plan（规划） → Implement（实施） → Commit（提交）
      ↑                                                      ↓
      └──────────────────── 下一轮 ←─────────────────────────┘
```

### 阶段详解

| 阶段 | 你该做什么 | AI 在做什么 | 推荐模式 |
|------|-----------|-------------|----------|
| **① Explore（探索）** | 告诉 AI 要改动的区域（如“我想给订单表加软删除”） | 读相关文件、grep 搜索、追踪引用链，理清现状 | **Plan Mode** |
| **② Plan（规划）** | 让 AI 产出详细修改方案，你审核方案的合理性与边界 | 生成具体计划、评估影响范围、列出风险点 | **Plan Mode** |
| **③ Implement（实施）** | 切出 Plan Mode，按已批准的计划执行 | 按顺序修改文件、运行构建、更新依赖 | Normal / Auto-Accept |
| **④ Commit（提交）** | 让 AI 生成 commit message，你确认后提交 | 生成规范的提交信息，可选自动发起 PR | Normal |

> **为什么要分阶段？**  
> 试想：你说“加个软删除功能”，15 分钟后 AI 改了 14 个文件，动了全局查询过滤器，破坏了 3 个现有接口——你只能手工回退。  
> 在 Plan Mode 里多花 5 分钟讨论方案，换来执行阶段节省 30 分钟返工。**规划即节省**。

## 三、项目文档与日志规范

为确保项目可追溯、可维护，请建立以下文档体系：

### 1. 开发日志（`logs/` 文件夹）
- 每天自动或手动记录：
  - **完成事项**：今日已完成的功能、修复的 Bug
  - **待办事项**：未完成的任务、下一步计划
- 文件名建议：`YYYY-MM-DD.md`，按日期归档。

### 2. 项目文档（`docs/` 文件夹）
存放所有与项目相关的规范文件，例如：

- `requirements.md` —— 需求文档（功能列表、用户故事）
- `technical.md` —— 技术选型、架构设计、数据库设计
- `design.md` —— UI/UX 设计规范、色彩体系、交互说明
- `process.md` —— 执行步骤、开发流程、代码规范
- `review-checklist.md` —— 代码审查清单

### 3. 上下文文件（`CLAUDE.md`）
在文件顶部或专门章节中，明确指引上述标准文件的路径及使用说明，例如：
```markdown
# 项目上下文

## 文档索引
- 需求 → [docs/requirements.md](docs/requirements.md)
- 技术设计 → [docs/technical.md](docs/technical.md)
- UI 规范 → [docs/design.md](docs/design.md)
- 开发流程 → [docs/process.md](docs/process.md)

## 工作说明
- 所有任务遵循 Explore→Plan→Implement→Commit 四阶段。
- 每日开发日志记录在 `logs/` 目录下。
- 代码提交前必须通过 `/review` 自动化测试。
```

## 四、提交需求模板（用于 Step 1 描述项目目标）

在项目初始化时，请按以下模板向 AI 描述需求，确保输出结果符合预期：

```markdown
# 项目需求提交模板

## 1. 项目概述
- **软件名称**：（如“任务管理助手”）
- **运行设备**：（如“Windows 桌面 / iOS 手机 / Web 浏览器”）

## 2. 功能目标
- **核心功能**：（如“创建任务、设置截止日期、分配负责人”）
- **解决的问题**：（如“帮助团队跟踪项目进度，减少遗漏”）

## 3. 页面与内容
- **页面列表**：（如“任务列表页、任务详情页、统计看板”）
- **每个页面的关键内容**：（列表页显示任务标题、状态、截止日期；详情页显示完整描述、评论等）

## 4. 设计风格与色彩
- **设计风格**：（如“极简/扁平/拟物/新拟态”）
- **主色**：（如“#2C3E50（深蓝）+ #E74C3C（珊瑚红）”）
- **辅助色**：（如“背景浅灰 #F5F7FA”）
```

将此模板内容填入项目初始化阶段的描述中，AI 将据此生成更精准的骨架代码和配置文件。

## 五、最佳实践总结

| 原则 | 说明 |
|------|------|
| **小步快跑** | 每次只处理一个功能，完成后立即提交，避免大变更积压。 |
| **先审后动** | 在 Plan Mode 中充分讨论方案，杜绝“边写边改”的混乱。 |
| **文档先行** | 在写代码前先更新 `docs/` 和 `logs/`，保持认知同步。 |
| **自动化测试** | 每次 review 都让 AI 生成测试并运行，确保不会破坏已有功能。 |
| **上下文延续** | 维护好 `CLAUDE.md`，让 AI 始终理解项目全貌。 |

遵循以上流程，你将从“救火队员”转变为“架构师”，让 AI 成为真正可靠的协作伙伴。
